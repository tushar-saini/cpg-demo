================================================================================
AI ENGINEER — ONE-DAY BUILD INTERVIEW
Project: Multi-Country Content Q&A with Citations
================================================================================

Welcome. This is a one-day practical exercise for AI engineering candidates.
You will spend roughly six hours building a working prototype from scratch,
commit it to a GitHub repository, and then walk a senior engineer through
your code and architecture for 30–45 minutes. The goal is not to produce a
polished product. The goal is to demonstrate that you can go from a plain
problem statement to a working, inspectable AI system in a single day.

Read this document once end-to-end before you start building. Ask any
clarifying questions in the 30-minute kickoff. Once you begin the build
block, you are on your own until the midday check-in.

--------------------------------------------------------------------------------
1. BUSINESS CONTEXT
--------------------------------------------------------------------------------

You are joining a team that builds the customer-domain platform for a
multi-country B2B retail business. Customers are small businesses that sell
products from multiple categories; the platform serves content (terms of
service, FAQs, promotional banners, static pages, product announcements)
through the web and mobile apps. Every customer sees content scoped to
their country and their preferred language.

Examples of the countries and languages currently in scope:

    Country A   English + Hindi
    Country B   English + Spanish
    Country C   English + French
    Country D   English

Content is authored by internal editors per country, stored in a central
database, and served through a REST API.

Today the app serves content by exact-match lookup (type + language +
country). There is no natural-language search. Customers who have questions
must either read through long documents or contact support. The team wants
a prototype of a smarter answer: given a natural-language question in the
customer's language, return a focused answer grounded in the official
content, with citations that point back to the exact source.

--------------------------------------------------------------------------------
2. YOUR GOAL
--------------------------------------------------------------------------------

Build a working end-to-end system that:

  (a) Ingests a sample corpus of content items and stores them in a vector
      database along with enough metadata to filter by country, language,
      and content type.

  (b) Exposes an HTTP endpoint that accepts a question, a country, and a
      language, and returns a grounded answer with citations pointing back
      to the source content items.

  (c) Uses a LangGraph-based agent to orchestrate the retrieval, filtering,
      synthesis, and citation steps. A single call to an LLM without a
      graph structure does not meet the requirement.

  (d) Enforces multi-tenant scoping. A question from Country A must never
      return an answer grounded in content authored for Country B, even if
      the Country B content is more semantically similar.

  (e) Returns citations the user (and the interviewer) can verify — the
      cited text must be present in the source content.

  (f) Ships as a GitHub repository with a README, a short architecture
      diagram, a requirements file, and a small evaluation harness.

You are not building a polished product. You are building a working
inspection target: a system we can run, break, and discuss.

--------------------------------------------------------------------------------
3. PROBLEM STATEMENT (what "done" looks like)
--------------------------------------------------------------------------------

At the end of the day, a reviewer should be able to:

  1. Clone your repository from GitHub
  2. Run a single command (documented in your README) that:
       - Starts the vector DB (local or containerized)
       - Ingests the sample content corpus
       - Starts the API server
  3. Send a POST request to your endpoint like this:

         POST /ask
         {
           "question": "Can I cancel my account if I have pending orders?",
           "country":  "B",
           "language": "es"
         }

  4. Receive a response like this:

         {
           "answer": "You can request account cancellation at any time,
                      but pending orders must be completed or cancelled
                      first. Contact support if you need help...",
           "language_used": "es",
           "citations": [
             {
               "content_id": "tc_2025_v3_es_sec_12",
               "type": "TERMS_AND_CONDITIONS",
               "excerpt": "Pending orders must be completed or cancelled
                           before an account can be closed...",
               "match_score": 0.87
             },
             {
               "content_id": "faq_account_closure_es",
               "type": "FAQ",
               "excerpt": "If you have active orders...",
               "match_score": 0.73
             }
           ],
           "trace": {
             "retrieval_count": 2,
             "latency_ms": 840,
             "model": "<name of the LLM you used>"
           }
         }

  5. Run the same question with country "A" (which has no Spanish content)
     and see either (a) a graceful fallback behavior you chose, or (b) an
     empty result — whichever you can defend. What you should NOT see is
     a Country B answer leaking through to a Country A caller.

  6. Run your evaluation harness and see at least a handful of test
     questions with expected answers (or expected citation IDs) and a
     simple pass/fail summary.

If all of this works, you have completed the exercise.

6. MULTI-TENANT SCOPING (read this carefully)
--------------------------------------------------------------------------------

This is the most important architectural point in the exercise. Every
content item in the corpus has a country and a language. The API caller
provides a country and a language at query time. Your system MUST NOT
return content authored for a different country, regardless of how
semantically similar it is.

Concrete expectations:

  - When you store a content chunk in the vector DB, you must also store
    {country, language, type, content_id, version} as metadata alongside
    the embedding.

  - When you retrieve, you must filter by the caller's country and
    language BEFORE (or during) the similarity search. Do not fetch top-K
    by similarity only and then filter — that leaves an information leak
    path through the retrieval budget.

  - If a caller asks a question in a language for which no content exists
    in their country, you must decide how to handle it. Three reasonable
    answers are: (a) return empty with a message, (b) fall back to another
    language in the same country and translate in the LLM step, (c)
    return the question unanswered with an error code. All three are
    acceptable — what matters is that you chose one deliberately and can
    explain why in the follow-up session if you are shortlisted.

  - If two countries happen to have content with overlapping IDs, your
    system must still distinguish them. Rely on the metadata, not the ID.

If you are shortlisted, we will probe this in the follow-up session
with an adversarial question. Be ready to defend your design.

    country           one of: A, B, C, D

countries (those are country-specific variants), and a few have
information in one country that is contradicted in another — your
multi-tenant scoping must handle those cases correctly.

9. EVALUATION RUBRIC (transparent)
--------------------------------------------------------------------------------

Scoring happens in two passes:

  Pass 1 (asynchronous, after 17:00 today): we clone your repository,
  run it, read the code and the README, and look at your screenshots.
  This is the initial screen — we decide who moves forward based on
  the submitted artifacts alone. No walk-through, no live demo.

  Pass 2 (follow-up session, only if shortlisted): we ask you to
  walk through the code live and answer the questions listed in
  section 10. Most of the scoring dimensions below are revisited
  with live probing.

We score on six dimensions. Each is scored 1–5. Dimensions (1)-(4)
are scored primarily from the repository and screenshots; dimensions
(5) and (6) are scored primarily from the follow-up session.

(1) Does it work end-to-end?
    Can we clone, run, and hit the endpoint? Does it return valid,
    grounded, cited answers for at least the sample questions? This is
    the floor — if this fails, the other dimensions do not matter much.

(2) Multi-tenant correctness
    Does country and language filtering actually work? Can we reproduce
    the guarantee that Country A never sees Country B content? Is the
    metadata stored with the vectors, or added after-the-fact?

(3) Citation fidelity
    Are the citations faithful? If the answer says "X is true" and the
    citation points to chunk Y, does chunk Y actually say X? Or is the
    LLM hallucinating and the citation is decorative?

(4) Architecture quality
    Is the code organized? Are concerns separated (ingest vs serve,
    agent vs API)? Is the LangGraph structure explicit and understandable?
    Are there obvious places you could swap in a different vector DB or
    LLM without rewriting everything?

(5) Production thinking
    When we ask "how does this scale" or "what fails at 100 QPS" or
    "what happens if the LLM is down", can you answer concretely? Did
    you make tradeoffs deliberately, or did you ignore production
    concerns entirely?

(6) Velocity and judgment
    Did you finish the core in 6 hours, or did you run out of time on
    yak-shaving (perfect Docker setup, exotic vector DB, custom
    embedding model)? Did you spend time on things that mattered, or
    on things that looked clever?

A strong candidate scores 4–5 on (1), (2), and (3), and 3–4 on the
other three. A hire scores 3+ on all six. Below 3 on (1) or (2) is a
fail regardless of the other scores.

10. FOLLOW-UP SESSION (if shortlisted)
--------------------------------------------------------------------------------

There is NO live walk-through today. After 17:00 you are done with the
build. We will review your repository asynchronously after the day
ends and will contact shortlisted candidates to schedule a separate
follow-up session within a few business days.

If you are invited to the follow-up session, come prepared to:

  - Share your screen and walk a senior engineer through your code
    and architecture in depth. The session is typically 45 to 60
    minutes.

  - Demonstrate the system running live from your machine, using the
    exact repo you submitted. If the environment you submitted does
    not run on the day of the follow-up, that is a serious problem —
    make sure setup is reliable before you push.

  - Answer detailed questions about your design choices. Expect
    something close to this list:

      1. Show me your LangGraph state graph. Walk me through what
         each node does and why you drew the edges the way you did.

      2. Show me the ingestion pipeline. What is your chunking
         strategy and why? What does one row in the vector DB
         actually look like?

      3. Show me how country and language filtering happens at query
         time. Walk me through exactly where in the code the filter
         is applied and why it is applied there and not later.

      4. Run a live query in English for Country A. Then the same
         question in Spanish for Country B. Walk through the
         response and the citations.

      5. Show a citation verification. Open the source content and
         find the quoted excerpt. Is it actually there?

      6. Handle an adversarial question we give you live: a question
         that should not have an answer, or in a language/country
         combination you did not test for. We are looking for
         graceful behavior, not perfect.

      7. What is your per-query cost? Per-query latency? How did
         you measure?

      8. If this went to production tomorrow with 100 queries per
         second, what breaks first? What would you change?

      9. If you had another full day, what is the single most
         impactful thing you would add, and why?

     10. If the LLM provider had to switch next week, how much code
         changes? Walk us through the blast radius.

  - Explain the structure of your repository and why you organized it
    the way you did. Be ready to justify every significant file and
    every dependency.

  - Confirm, in your own words, that the code is yours and that you
    understand every line of it — including any parts you wrote with
    the help of an LLM or coding assistant. We will ask about specific
    lines. If you cannot explain something you committed, that is
    information for us about how the solution actually came together.

The purpose of the follow-up session is not to trick you. The purpose
is to confirm you built what you submitted, you understand how it
works, and you can reason about it under moderate pressure. A
candidate who builds a good system but cannot explain it is not the
candidate we are hiring.

You do not need to rehearse answers. We want to see how you think out
loud. Uncertainty is fine — "I don't know, but here is how I would
find out" is a better answer than a confident wrong guess.

13. FAQ / PRACTICAL TIPS
--------------------------------------------------------------------------------

Q: Can I use an LLM to help write the code?
A: Yes. We expect you to. Modern AI engineering is AI-assisted. Use any
   tools you normally use (Copilot, Cursor, IDE integrations, an LLM as
   a coding partner). What we score is the final result — does it work,
   is it sound, can you explain it, and can you defend it in the
   follow-up session if shortlisted. A candidate who used AI to move
   faster and understood every line is a better signal than one who
   wrote everything by hand but produced less.

Q: Do I need to bring my own vector DB?
A: Yes. Use whatever runs on your laptop (Chroma, FAISS) or spin up a
   local container (pgvector in Docker, Qdrant). We do not provide
   hosted vector DB access. A local one is fine and is usually faster
   to iterate on during the build.

Q: Do I need to bring my own LLM API key?
A: YES. We do NOT provide API keys or hosted LLM access for this
   exercise. Bring your own Anthropic or OpenAI key with enough
   credits to run through the full build today (and to run the demo
   again later if you are shortlisted for the follow-up session). A
   full day of iterative development plus a demo is typically a few
   dollars of spend. If you do not have a key on the day of the
   kickoff, tell us — but the default expectation is that a serious
   AI candidate already has keys. We will not extend the day to let
   you sign up for a new API account.

Q: Do I need to bring my own GitHub?
A: YES. Create a repository on your own GitHub account (public, or
   private with view access granted to the interviewer). You will
   push your work there throughout the day and share the link at
   17:00 when the build window closes. This is part of the exercise
   — we want to see how you organize a repo from scratch.

Q: What if my approach is unusual?
A: That is fine. Explain it in the README and be ready to defend it
   in the follow-up session if you are shortlisted.

Q: I finished early. Can I keep adding features?
A: Yes, but prioritize polish over novelty. A well-documented, well-tested
   6-hour build scores higher than a sprawling 8-hour build with broken
   edges.

Q: What if the evaluation harness fails on some questions?
A: That is fine, as long as you can explain why. Do not hide failing
   cases — document them. We would rather see an honest 6/10 pass rate
   with a clear explanation than a suspicious 10/10.

Q: Do I need to build a UI?
A: No. The API is enough. A curl command in the README is fine.

Q: Can I ask for help during the build?
A: Clarifying questions about the problem: yes, anytime. Debugging help
   or design advice: no — that is what we are testing.

APPENDIX A — SAMPLE CORPUS
================================================================================

The following is the sample content corpus for this exercise. 44 items
across 4 countries, 4 languages, 5 content types.

Each item is one JSON object on one line (JSONL format). Copy this
entire section (everything between the "BEGIN CORPUS" and "END CORPUS"
markers below) into a file called corpus.jsonl in your repository. You
can load it with:

    import json
    with open("corpus.jsonl") as f:
        items = [json.loads(line) for line in f if line.strip()]

Country codes in the corpus:
  A = English + Hindi
  B = English + Spanish
  C = English + French (Canadian)
  D = English only

Important notes about the corpus:

  * Some topics (account closure, returns, payment) are covered across
    multiple countries with DIFFERENT rules per country. Do NOT let
    content from Country A answer a question from Country B, even if
    the semantic similarity is high.

  * Hindi, Spanish, and French strings are short and simplified; they
    are representative, not literary translations.

  * Two items (Country B FAQ b_faq_returns_en and Country A FAQ
    a_faq_returns_en) have contradictory return windows. If your
    system confuses them across countries, we will notice.

  * The Country D corpus is deliberately small — test your system
    against a country that has sparser content too.
